## Context

当前主界面入口已经能够展示默认关卡并发出 `StartLevelRequestedEvent`，但默认关卡标识、名称和描述仍位于 `MainModel` 代码常量中。第一阶段后续的关卡运行时上下文、波次推进和非战斗波次占位，需要先拥有稳定的配置表结构，才能在实现时从配置读取关卡和波次定义。

项目已有 Luban 生成的 `GameConfig.Tables` 和 `Assets/AssetRaw/Configs/bytes/item_tbitem.bytes`，但配置源数据位于仓库外的 `D:\UnityGame\Self\RogueCard\Configs\GameConfig\Datas`，当前 Unity 工程内没有现成关卡表源文件。

## Goals / Non-Goals

**Goals:**

- 在配置源数据目录中新增第一阶段所需的关卡主表结构。
- 在配置源数据目录中新增第一阶段所需的关卡波次表结构。
- 新增波次类型枚举，覆盖战斗、宝箱和商店三类节点。
- 只定义字段结构，不要求填充完整玩法数据。
- 让后续关卡运行时上下文可以通过关卡标识定位关卡配置，并通过波次列表或波次顺序定位当前波次。

**Non-Goals:**

- 不生成 C# 配置代码或 bytes 资源。
- 不接入 Unity 运行时加载逻辑。
- 不实现 `LevelRuntimeContext`、波次推进、战斗流程、宝箱奖励或商店购买。
- 不新增怪物、卡牌、奖励、玩家初始状态、商店商品等第二阶段或第三阶段配置表。

## Decisions

### 1. 使用两张表承载第一阶段关卡骨架

新增 `TbLevel` 和 `TbLevelWave` 两张表。`TbLevel` 描述关卡级展示信息、默认入口标记和波次列表；`TbLevelWave` 描述每个波次的所属关卡、顺序、类型和占位展示文案。

- 选择原因：第一阶段只需要入口、上下文和波次分发，两个表可以覆盖全部需求，同时避免提前设计战斗遭遇、宝箱奖励和商店商品。
- 替代方案：把波次信息内联在关卡表中。该方式表数量更少，但后续维护波次、复用波次或为波次增加独立负载字段时会更难扩展。

### 2. 使用 `WaveType` 枚举限制波次类型

`TbLevelWave.Type` 使用枚举值，首版只包含 `Battle`、`Chest`、`Shop`。

- 选择原因：波次推进需要稳定分发类型，枚举比字符串更容易在生成代码中约束合法值。
- 替代方案：使用字符串类型字段。该方式录表更自由，但容易产生拼写错误，后续运行时还需要额外校验。

### 3. 第一阶段保留 `PayloadId` 作为空负载预留

`TbLevelWave` 保留可为空的 `PayloadId` 字段，第一阶段可以不填。后续战斗遭遇、宝箱奖励或商店池配置完成后，再由不同波次类型解释该字段。

- 选择原因：可以让波次表结构在第一阶段和后续阶段之间保持稳定，避免刚接入波次推进后立即改表。
- 替代方案：为每种波次类型提前创建独立外键字段。该方式类型更明确，但会提前引入尚未设计的表。

### 4. 暂不创建 `TbLevelEntry`

默认关卡入口通过 `TbLevel.IsDefault` 表达，不单独创建入口表。

- 选择原因：当前只有一个默认关卡入口，单独入口表会增加同步成本。
- 替代方案：创建 `TbLevelEntry` 管理主界面展示列表。该方式适合多关卡选择，但超出第一阶段最小闭环范围。

## Risks / Trade-offs

- [风险] 仓库外配置目录不受当前 Unity 工程 git 状态约束 → [缓解] 实施时明确列出新增文件路径，并在生成配置前确认配置仓库或目录的版本管理方式。
- [风险] 只建字段不填数据会导致后续运行时找不到默认关卡 → [缓解] 本变更只负责表结构，后续接入运行时前应补一行 `level_default_001` 和对应波次数据。
- [风险] `PayloadId` 是弱类型字段，后续可能出现类型解释错误 → [缓解] 在各波次类型真正接入时增加对应规格和校验。
