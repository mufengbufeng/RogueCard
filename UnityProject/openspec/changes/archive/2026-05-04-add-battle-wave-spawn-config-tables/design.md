## Context

当前 `add-level-wave-config-tables` 已定义 `TbLevelWave`，并为战斗、宝箱和商店波次保留 `PayloadId`。战斗波次后续需要表达“分批刷怪，前一批全部死亡后再进入下一批”的数据，但项目目前还没有怪物运行时、死亡事件或战斗波次推进实现。

本设计只补齐配置层结构：让战斗波次可以通过 `PayloadId` 指向一个刷怪方案，再由刷怪方案列出多个有序批次。运行时如何监听死亡、生成实体和完成波次，留给后续独立变更实现。

## Goals / Non-Goals

**Goals:**

- 定义最小怪物配置结构，提供稳定怪物标识和占位资源标识。
- 定义战斗波次刷怪方案配置结构，作为 `TbLevelWave.PayloadId` 的目标。
- 定义刷怪批次配置结构，表达某个刷怪方案内的批次顺序、怪物和数量。
- 保持配置结构足够简单，支持首版“上一批死完再刷下一批”。

**Non-Goals:**

- 不实现运行时刷怪推进逻辑。
- 不实现怪物实体、血量、伤害、死亡事件或战斗结算。
- 不配置正式怪物数值、AI、技能、掉落、奖励或卡牌联动。
- 不生成 Luban C# 代码或 bytes 资源。
- 不修改宝箱、商店、关卡入口或 UI 运行时逻辑。

## Decisions

### 1. 使用独立刷怪方案表承接 `PayloadId`

新增 `TbBattleWaveSpawn` 表，`TbLevelWave.PayloadId` 在 `Battle` 类型下指向该表的 `Id`。该表记录刷怪方案标识、名称、说明和批次列表。

- 选择原因：`PayloadId` 已是波次表预留负载字段，使用独立表可以让战斗波次负载和宝箱、商店后续负载保持一致的扩展方式。
- 替代方案：直接让 `TbLevelWave.PayloadId` 指向第一批刷怪批次。该方式更少一张表，但无法自然表达一个战斗波次包含多个批次，也不利于复用或校验完整方案。

### 2. 使用批次表表达顺序刷怪

新增 `TbBattleWaveSpawnBatch` 表，每行描述一个批次：所属刷怪方案、顺序、怪物标识和数量。运行时后续可按 `Order` 从小到大读取，当前批次怪物全部死亡后进入下一批。

- 选择原因：行式结构比把多个批次塞进一列更清晰，后续增加出生点、间隔、阵位或权重时可以扩展字段。
- 替代方案：在刷怪方案表内使用列表 bean 表达所有批次。该方式文件更少，但多批次内容可读性和后续扩展性较差。

### 3. 先定义最小怪物表，不引入完整怪物数值

新增 `TbMonster` 表，只包含怪物标识、名称、描述和资源标识。首版刷怪配置只需要知道“刷什么”和“生成几个”。

- 选择原因：刷怪批次需要稳定引用目标，但完整怪物属性、血量和战斗行为属于后续战斗系统范围。
- 替代方案：批次表直接填写资源路径，不建怪物表。该方式更快，但后续补怪物属性时会产生字段迁移。

### 4. 不填正式玩法数据

本变更只建立表头和必要字段，不要求填入正式刷怪方案、怪物或批次数据。

- 选择原因：当前目标是先形成可生成的配置结构，避免在怪物系统尚未确定时提前承诺数值内容。
- 替代方案：直接填一套默认怪物和默认战斗波次。该方式便于后续运行时联调，但会把数据设计和结构设计混在一起。

## Risks / Trade-offs

- [风险] `TbLevelWave.PayloadId` 是弱类型字段，可能错误指向非刷怪方案 → [缓解] 后续运行时读取 `Battle` 波次时只按 `TbBattleWaveSpawn` 解释，并在接入变更中增加校验。
- [风险] 只建结构不填数据会导致默认关卡运行时仍无法刷怪 → [缓解] 在任务中记录后续接入前需要补充默认刷怪方案和批次数据。
- [风险] 怪物表字段过少，后续战斗属性需要扩展 → [缓解] 本阶段只承诺最小引用结构，后续战斗系统变更可追加属性字段。
