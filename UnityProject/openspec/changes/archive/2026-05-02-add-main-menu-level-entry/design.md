## Context

项目启动链路已经裁剪为 `GameEntry` 加载 HotFix 后进入 `GameLogicEntry`，再由 `InitProcedure` 切换到 `MainMenuProcedure` 打开 `MainView`。现有主界面控制器可以在缺少示例玩法模块时正常初始化，但 `HandleStartGame` 仍然只显示“请接入正式游戏逻辑”的占位反馈。

卡牌 Rogue 最小闭环的第一阶段需要一个稳定入口：主界面先展示默认关卡，点击开始后产生可被后续关卡运行时上下文消费的请求。本变更只负责入口请求，不直接实现关卡流程或波次推进。

## Goals / Non-Goals

**Goals:**

- 让主界面显示默认关卡的最小信息，确保用户知道点击开始会进入哪个关卡。
- 让开始按钮发出默认关卡进入请求，并携带稳定的默认关卡标识。
- 保持主界面在关卡系统尚未实现时仍可独立运行和测试。
- 把需要用户在 Unity 编辑器中确认的 Prefab/绑定工作与 AI 可完成的代码工作分离。

**Non-Goals:**

- 不实现 `LevelProcedure`、关卡运行时上下文、波次推进或关卡结果。
- 不新增 Luban 关卡表，不把默认关卡数据接入正式配置生成流程。
- 不制作或精修 Unity Prefab，不调整复杂 UI 布局。
- 不实现战斗、宝箱、商店、奖励、地图路线或存档。

## Decisions

### 1. 默认关卡信息先由主界面代码提供

主界面模型保存默认关卡标识、名称和描述，供 View 展示和 Controller 发起请求时使用。

- 选择原因：当前工程只有 `item_tbitem.bytes`，没有现成关卡表；第一阶段入口不应被配置生成流程阻塞。
- 替代方案：立即新增 Luban 关卡表。该方案更接近最终形态，但会把表结构、生成命令、资源收集和加载路径都纳入本变更，范围过大。

### 2. 开始按钮只发出请求，不直接承担关卡流程实现

`MainController` 在点击开始时构造默认关卡请求，并通过轻量事件或等价机制对外发布。后续 `add-level-runtime-context` 或 `add-level-wave-progression` 变更再消费该请求并切换流程。

- 选择原因：主界面入口和关卡运行时是两个可独立验收的模块，解耦后可以先测试按钮语义和请求数据。
- 替代方案：点击按钮直接切换到新关卡流程。该方案更直观，但会迫使本变更同时创建并注册关卡流程，越过第一阶段拆分边界。

### 3. 未接入关卡流程时显示“请求已发出”的正向反馈

在后续关卡流程尚未实现前，按钮点击后的用户反馈应表达默认关卡请求已提交，而不是继续显示未接入玩法逻辑。

- 选择原因：这能让当前变更被独立手动验证，同时避免误导用户认为按钮仍是旧占位。
- 替代方案：点击后不显示任何反馈，只发事件。该方案更纯粹，但在 Unity Play 验证时不直观。

### 4. Prefab 工作作为用户验收项，不作为 AI 代码任务

AI 负责提供字段、数据和测试；用户在 Unity 编辑器中确认 `MainView.prefab` 的按钮与文本绑定、布局和资源地址。

- 选择原因：Prefab 制作和视觉布局依赖 Unity 编辑器交互，AI 容易产生 YAML 级误改。
- 替代方案：AI 直接修改 Prefab YAML。该方案可自动化，但风险较高，且不适合首阶段 UI 骨架验证。

## Risks / Trade-offs

- [风险] 默认关卡信息硬编码后未来需要迁移到配置表 → [缓解] 本变更只定义最小默认数据，后续配置表变更再替换数据来源，保持默认关卡标识稳定。
- [风险] 请求发布机制与后续关卡流程消费方式不一致 → [缓解] 请求数据保持轻量，只要求包含默认关卡标识，后续可在不改变 UI 展示语义的情况下调整消费端。
- [风险] Prefab 缺少文本绑定导致运行时展示不完整 → [缓解] 保留现有运行时文本兜底，并把 Unity 编辑器绑定确认列入用户完成任务。
- [风险] 本变更被误解为要进入真实关卡 → [缓解] specs 和 tasks 明确只验收请求发出，不验收关卡流程执行。
