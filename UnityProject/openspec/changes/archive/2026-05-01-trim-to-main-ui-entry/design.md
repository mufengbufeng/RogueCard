## Context

当前启动链路为 `GameEntry -> GameLogicEntry -> InitProcedure -> LoginProcedure -> MainMenuProcedure`，并通过 `EntryView` 进入 `MainView`，`MainView` 再继续衔接体力、关卡和玩法流程。这条链路把框架示例中的业务逻辑和项目骨架耦合在一起，不利于将当前仓库作为新项目模板继续开发。

本次设计需要在不破坏 Runtime/HotFix 分层、EF 管理器注册和 UI 系统基础能力的前提下，将可见入口压缩为单一主页，并移除所有对示例玩法可运行性的依赖。

## Goals / Non-Goals

**Goals:**
- 保留 `GameEntry` 与 `GameLogicEntry` 的启动骨架不变，仍然通过热更新入口完成初始化。
- 将流程收敛为可直接打开 `MainView` 的最小路径，不再要求经过 `EntryView` 或切换到玩法流程。
- 让主界面在没有体力、关卡、战斗模块的情况下也能稳定显示，作为项目后续开发入口。
- 清理已失去作用的示例流程、控制器、模块接线和相关测试，避免残留死代码与失效依赖。

**Non-Goals:**
- 不在本次变更中设计新的正式首页视觉或交互。
- 不重构 EF 框架层管理器、ModuleSystem 或 HybridCLR 加载机制。
- 不保留示例战斗逻辑的可运行能力。

## Decisions

### 1. 保留 Runtime/HotFix 启动骨架，只裁剪 HotFix 业务入口后的流程
- 选择原因：`GameEntry` 与 `GameLogicEntry` 已承担框架初始化和热更装配职责，是项目模板的核心资产；本次目标是去掉示例业务，而不是改造底层框架。
- 备选方案：直接把主界面启动前移到 Runtime 层。未采用，因为会打破现有两层分离约定，并让后续热更新 UI 逻辑回迁成本更高。

### 2. 以 `MainView` 作为唯一入口，移除 `EntryView -> MainView` 双层路径
- 选择原因：仓库里现有主界面资源与控制器已经齐备，直接复用 `MainView` 比保留单独的 `EntryView` 更接近“保留一个主界面入口”的目标。
- 备选方案：保留 `EntryView` 作为占位首页。未采用，因为它本质仍是二级跳转壳层，会继续保留多余流程与控制器。

### 3. 将 `MainController` 降级为不依赖体力、关卡、玩法流程的占位控制器
- 选择原因：当前 `MainController` 强依赖 `IEnergyModule`、`ILevelModule` 和 `MainMenuProcedure.StartGame()`，若只删除玩法流程会留下大量不可用分支。直接把主界面收敛为“可展示、可响应基础按钮/文案”的主页，可最大限度降低耦合。
- 备选方案：保留体力和关卡模块但禁止进入玩法。未采用，因为这仍然保留了示例游戏规则，不符合“清理掉框架游戏逻辑”的目标。

### 4. 示例玩法相关代码优先从活跃入口链中移除，再视引用关系决定是否物理删除
- 选择原因：先保证启动链、流程链和 UI 入口干净，再处理玩法目录、测试和资源，可避免一次性大删导致编译与引用关系难以定位。
- 备选方案：一次性删除所有 `GamePlay`、`Energy`、`Level` 目录。未采用，因为需要先确认 asmdef、测试和其他引用是否已完全脱钩。

## Risks / Trade-offs

- [主界面 Prefab 仍包含示例文案或控件命名] → 通过占位文案和最小交互验证，确保它仍然可以作为新项目主页使用。
- [删除流程或模块后产生编译引用残留] → 先收敛入口，再逐步清理控制器、测试与玩法目录，并通过全局引用搜索验证。
- [现有测试覆盖了被移除的示例逻辑] → 同步删除或改写相关 EditMode 测试，只保留仍有意义的框架测试。
- [主界面过度简化导致后续扩展点不清晰] → 保留 `MainView/MainController/MainModel` 这组 UI 骨架，作为后续真实业务接入点。
