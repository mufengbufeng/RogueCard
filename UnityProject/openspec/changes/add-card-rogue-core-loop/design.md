## Context

项目目标仍然是做出卡牌 Rogue 最小可玩闭环：从主界面进入默认关卡，按波次推进战斗、宝箱和商店节点，在战斗中使用卡牌击败 3x3 阵型怪物，并通过经验升级获得三选一卡牌奖励。

这个目标会涉及多个较大的玩法模块。为了降低单个 OpenSpec 变更的范围和验收压力，本变更只保留总体路书；后续每个模块都需要独立立项、独立规格、独立任务和独立验证。

## Goals / Non-Goals

**Goals:**

- 明确卡牌 Rogue 最小闭环的模块拆分顺序。
- 明确每个后续模块的范围边界，避免一次性实现所有系统。
- 作为后续 OpenSpec 变更的任务看板，记录哪些模块已经单独立项和完成。
- 保持当前变更对代码零影响。

**Non-Goals:**

- 不在本变更中实现主界面入口、关卡流程、战斗系统、怪物行为、奖励系统或 UI。
- 不在本变更中新增或修改 delta spec。
- 不在本变更中新增配置表、测试、Prefab 或运行时代码。
- 不把宝箱经济、商店购买、遗物、地图路线、存档和多角色纳入最小路书。

## Decisions

### 1. 本变更只作为路书看板

`add-card-rogue-core-loop` 只描述整体路线和模块拆分，不直接进入代码实现。

- 选择原因：核心闭环范围过大，一次性实施会导致任务难以验收，也不利于后续针对单个模块调整设计。
- 替代方案：保留原来的 4 个模块规格并一次性实施。该方式能更快形成大任务，但会让 `/opsx:apply` 承载过多系统，风险更高。

### 2. 每个模块单独创建 OpenSpec 变更

后续模块开工时，应单独创建如 `add-main-menu-level-entry`、`add-level-wave-progression`、`add-monster-grid-behavior`、`add-card-battle-core` 等变更。

- 选择原因：每个模块都可以拥有独立 proposal、spec、design、tasks 和验证记录。
- 替代方案：在一个变更内用 tasks 拆分所有模块。该方式缺少模块级规格边界，后续难以归档和追踪。

### 3. 先搭骨架，再做战斗，再做奖励和表现

建议实施顺序为：主界面入口与关卡骨架 → 关卡波次推进 → 怪物阵型规则 → 卡牌战斗回合 → 经验升级与三选一 → 宝箱/商店占位 → UI 与验证打磨。

- 选择原因：入口和关卡流程是承载后续玩法的外壳；战斗和怪物规则是核心；奖励和 UI 可在核心规则稳定后接入。
- 替代方案：先做完整战斗再接关卡。该方式便于规则测试，但可能导致战斗上下文与关卡推进后续再重接。

## Risks / Trade-offs

- [风险] 路书看板被误当成可直接实施变更 → [缓解] proposal、design 和 tasks 都明确写明本变更不直接修改代码。
- [风险] 拆分过细导致管理成本上升 → [缓解] 只按可独立验收的模块拆分，不为单个小类或临时实现单独立项。
- [风险] 后续模块顺序随实现发现变化 → [缓解] 本路书允许调整顺序，以独立 OpenSpec 变更中的最新设计为准。
- [风险] 删除原模块 specs 后缺少细节 → [缓解] 后续模块立项时重新生成更聚焦的 specs。

## Open Questions

- 主界面入口是否应先完成到可进入空关卡，还是等待关卡波次骨架一起完成？建议先做可进入空关卡，减少第一步范围。
- 宝箱和商店是否作为一个占位模块，还是分别拆成两个模块？建议首版合并为“非战斗波次占位”。
- 三选一卡牌奖励是否先只服务升级，还是同时服务宝箱？建议先服务升级，宝箱复用作为后续节点。
