## Context

项目当前使用 Unity 6000.3、HotFix 游戏逻辑和 EF MVC UI 系统，已有主界面入口与关卡/波次配置相关工作，但局内还缺少统一的场景承载方式。局内表现需要同时满足两类需求：背景、怪物、受击特效需要自然的 2D 世界空间层级；手牌、按钮、玩家状态和奖励弹窗需要稳定的 UI 交互。

因此本变更把局内拆为 `Game` 场景、2D 战斗舞台和 `GameView` UI 三个协作部分：场景负责容器，舞台负责世界表现，GameView 负责玩家操作空间。

## Goals / Non-Goals

**Goals:**

- 创建 `Game` 场景，作为局内运行时的统一承载场景。
- 创建 `GameView`/`GameController` 局内 UI 骨架，并接入现有 EF UI MVC 约定。
- 创建战斗舞台预制体结构，包含背景层、怪物槽位层、世界特效层和前景层。
- 使用 2D 世界空间表现背景滚动、怪物站位和前后遮挡。
- 使用 UI 表现玩家状态、手牌区、目标线、按钮和后续弹窗承载点。
- 为后续怪物阵型、卡牌出牌、波次推进和奖励 UI 提供稳定挂载点。

**Non-Goals:**

- 不实现完整战斗规则、怪物 AI、卡牌效果、伤害结算或经验奖励。
- 不制作正式美术资源，只允许使用占位 Sprite、占位 Prefab 或基础 UI 元素。
- 不改变现有 Luban 表结构。
- 不把宝箱、商店、奖励选择的完整 UI 纳入本变更。
- 不引入新的渲染、动画或 UI 第三方依赖。

## Decisions

### 1. 局内采用“2D 战斗舞台 + UI 操作层”

背景、怪物、前景和世界特效使用 2D 世界空间对象；玩家状态、手牌、按钮、拖拽层和弹窗使用 UI。

- 选择原因：怪物前后层级、背景滚动、粒子特效和镜头反馈在 2D 世界空间更自然；卡牌拖拽和按钮交互在 UI 中更稳定。
- 替代方案：全 UI。该方案实现初期较快，但怪物遮挡、世界特效、镜头感和背景视差会变得别扭。
- 替代方案：全 2D。该方案适合动作游戏舞台，但卡牌手牌、弹窗、拖拽和布局适配成本更高。

### 2. `Game` 场景只作为局内容器

`Game` 场景内保留局内根节点、相机、舞台挂载点和 UI 打开入口，不在场景内硬编码具体关卡数据、怪物数据或卡牌数据。

- 选择原因：关卡内容后续会由配置和运行时上下文驱动，场景保持通用可以避免每个关卡复制 Scene。
- 替代方案：每个关卡一个场景。该方案直观，但会增加场景维护和资源切换成本。

### 3. 战斗舞台以 Prefab 承载环境主题

`BattleStage` 预制体提供固定根节点：`BackgroundRoot`、`MonsterSlotRoot`、`WorldFxRoot`、`ForegroundRoot`。后续可以按主题扩展为森林、洞穴、遗迹等不同舞台 Prefab。

- 选择原因：Prefab 比 Scene 更适合按波次或关卡配置加载和替换，也便于 YooAsset 资源化。
- 替代方案：直接把所有节点放在 `Game` 场景。该方案短期简单，但会让场景变成具体内容容器，降低复用性。

### 4. 怪物槽位使用固定 3x3 锚点

战斗舞台提供 9 个怪物槽位锚点，按后排、中排、前排划分。表现层通过锚点位置、缩放和排序表现前后关系，逻辑层仍只关心 3x3 槽位索引。

- 选择原因：后续怪物阵型、死亡前移、转圈行为和合法目标都需要稳定槽位模型；表现层固定锚点可以让动画和 UI 指向更可控。
- 替代方案：运行时动态计算所有槽位。该方案灵活，但首版调参和动画挂点成本更高。

### 5. 背景滚动表现只服务“前进过渡”

背景分为远景、中景、近景三层，前进时播放短暂滚动动画，不要求玩家或摄像机真实位移。

- 选择原因：卡牌 Rogue 的局内重点是节点推进和战斗决策，短过渡足以表达前进感，避免引入真实地图移动系统。
- 替代方案：真实摄像机移动。该方案沉浸感更强，但会牵涉场景长度、碰撞、镜头边界和更多资源制作。

### 6. GameView 只做局内 UI 骨架和表现承载

`GameView` 提供玩家状态区、手牌区、拖拽层、目标线层、回合按钮和弹窗根节点；`GameController` 负责打开窗口和后续连接局内模型，但本变更不实现完整战斗数据同步。

- 选择原因：先确定 UI 结构，后续战斗规则和卡牌系统可以逐步把数据接入这些区域。
- 替代方案：等战斗规则完成后再做 UI。该方案减少返工，但会让前期无法验证场景、舞台和操作区的整体比例。

## Risks / Trade-offs

- [风险] 2D 舞台和 UI 操作层需要坐标转换 → [缓解] 在舞台和怪物 Actor 上预留目标锚点，后续目标线和拖拽逻辑只对接锚点。
- [风险] 首版 Prefab 结构可能和后续正式美术资源不完全匹配 → [缓解] 只固定根节点和锚点契约，不把具体 Sprite 层级设计得过细。
- [风险] `Game` 场景过早承载玩法逻辑会导致后续难拆 → [缓解] 场景只保留容器和入口，玩法状态放在 HotFix 层模型或后续运行时上下文。
- [风险] 背景滚动如果做成真实移动会扩大范围 → [缓解] 本变更只要求短动画过渡，不引入地图移动。
- [风险] GameView 空骨架可能缺少可见反馈 → [缓解] 使用占位文本、按钮和手牌占位节点确保进入场景后可人工验证布局。
